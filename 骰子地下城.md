下面是一份**非官方的《骰子地下城 / Dicey Dungeons》游戏设计拆解文档**，适合用于开发参考、竞品分析或制作类似“骰子 + 卡牌/装备 + Roguelike”的轻量策略游戏。

> 说明：这不是官方 GDD，也不包含原作素材、关卡表、代码或商业机密。以下内容是基于公开玩法体验进行的系统化拆解。若用于开发，建议做差异化设计，避免直接复刻角色、美术、文案、装备名称和关卡内容。

# 《骰子地下城》游戏设计参考文档

## 1. 游戏概述

### 1.1 基本信息

| 项目 | 内容 |
|---|---|
| 游戏类型 | 回合制策略 / 骰子构筑 / Roguelike / 轻量卡牌 |
| 核心玩法 | 掷骰子，将骰子分配到装备槽中触发效果，击败敌人并推进地牢 |
| 单局时长 | 约 20-40 分钟 |
| 操作方式 | 鼠标 / 触屏 / 手柄均适配 |
| 目标体验 | 轻松、直观、随机、有策略、失败成本低 |
| 主要平台 | PC、主机、移动端均适合 |

### 1.2 一句话概括

玩家扮演被变成骰子的冒险者，在每回合投掷骰子，并将骰子放入不同装备中发动攻击、防御、治疗或特殊效果，通过随机奖励和路线选择构筑角色能力，最终挑战 Boss。

# 2. 设计目标

## 2.1 核心设计目标

1. **低门槛**
   - 玩家只需要理解：
     - 骰子点数
     - 装备槽条件
     - 敌我生命值
   - 不依赖复杂卡牌关键词。

2. **高可读性**
   - 每个装备的条件和效果必须一眼能看懂。
   - 战斗中玩家主要决策是：
     - 哪个骰子放到哪个装备？
     - 先用哪个效果？
     - 要不要重掷、拆分、变点？

3. **短局高反馈**
   - 一局不长，适合碎片时间。
   - 每场战斗都有奖励或推进感。
   - 失败后也能快速再开。

4. **构筑感**
   - 虽然没有传统卡组抽牌，但通过装备、升级、角色机制形成 Build。
   - 玩家会形成“这局我走毒伤流 / 冰冻流 / 大点数爆发流 / 多骰连击流”的策略。

5. **随机但可控**
   - 骰子随机提供不确定性。
   - 装备条件、重掷、改变点数、复制骰子等机制提供控制手段。

# 3. 核心体验支柱

## 3.1 骰子作为资源

传统卡牌游戏中，资源可能是法力值、费用、行动点。  
《骰子地下城》的核心创新是：

> 骰子既是随机资源，也是行动触发器。

玩家每回合获得若干个骰子，例如：

$$
D = \{d_1, d_2, d_3, ..., d_n\}
$$

其中每个骰子的点数：

$$
d_i \in \{1,2,3,4,5,6\}
$$

装备对骰子有不同要求：

| 条件类型 | 示例 |
|---|---|
| 任意点数 | 放入任意骰子，造成该点数伤害 |
| 偶数 | 只能放入 $2,4,6$ |
| 奇数 | 只能放入 $1,3,5$ |
| 最大值限制 | 只能放入 $1-3$ |
| 最小值限制 | 只能放入 $4-6$ |
| 指定点数 | 只能放入 $6$ |
| 累计槽 | 多次放入骰子，总和达到阈值后触发 |
| 双骰槽 | 需要两个骰子共同触发 |
| 成对要求 | 需要两个相同点数 |
| 点数递减/递增 | 需要满足顺序条件 |

# 4. 游戏循环

## 4.1 单局大循环

```text
选择角色
    ↓
进入地牢第 1 层
    ↓
选择路线 / 遭遇敌人
    ↓
回合制战斗
    ↓
获得经验 / 金币 / 装备 / 升级
    ↓
继续探索更深层
    ↓
挑战小 Boss / 精英 / 最终 Boss
    ↓
胜利或失败
    ↓
解锁新角色、新章节、新规则
```

## 4.2 战斗小循环

```text
玩家回合开始
    ↓
投掷骰子
    ↓
玩家分配骰子到装备
    ↓
装备触发效果
    ↓
玩家结束回合
    ↓
敌人投掷骰子
    ↓
敌人使用技能
    ↓
状态结算
    ↓
进入下一回合
```

## 4.3 回合决策结构

玩家每回合主要面临 4 类决策：

1. **资源分配**
   - 高点骰子用于伤害还是防御？
   - 低点骰子是否用于辅助装备？

2. **执行顺序**
   - 先重掷还是先攻击？
   - 先施加状态还是先造成伤害？

3. **风险收益**
   - 用不稳定装备赌高收益？
   - 留资源防御还是全力攻击？

4. **长期构筑**
   - 当前装备是否适合后续升级？
   - 是否替换旧装备？
   - 是否围绕某种状态流派构筑？

## 5.2 骰子系统

### 5.2.1 基础规则

每回合开始时，玩家获得指定数量骰子。

```text
roll_dice(count):
    dice = []
    for i in range(count):
        dice.append(random(1, 6))
    return dice
```

### 5.2.2 骰子变化机制

为了降低纯随机带来的挫败感，需要提供控制骰子的方式：

| 机制 | 说明 |
|---|---|
| 重掷 | 重新投掷一个或多个骰子 |
| 加点 | 使骰子点数 $+1$ |
| 减点 | 使骰子点数 $-1$ |
| 翻面 | 将点数变为 $7 - x$ |
| 复制 | 复制一个骰子 |
| 拆分 | 将一个高点数骰子拆成两个低点骰子 |
| 合并 | 将两个骰子合成一个更高点数 |
| 锁定 | 下回合保留该骰子 |
| 固定点数 | 直接生成指定点数骰子 |

例如翻面：

$$
d' = 7 - d
$$

如果骰子是 $2$，翻面后变为 $5$。

## 5.3 装备系统

装备是本游戏的“卡牌”。  
玩家不是抽卡，而是携带固定装备进入战斗。

### 5.3.1 装备基础属性

| 字段 | 说明 |
|---|---|
| 装备 ID | 唯一标识 |
| 名称 | 显示名称 |
| 尺寸 | 占用装备栏格数 |
| 使用次数 | 每回合可使用次数 |
| 骰子条件 | 可接受什么骰子 |
| 效果 | 造成伤害、防御、治疗、状态等 |
| 是否可升级 | 是否有强化版本 |
| 稀有度 | 普通、稀有、特殊 |
| 标签 | 攻击、防御、毒、火、冰、电等 |

示例：

```json
{
  "id": "basic_sword",
  "name": "短剑",
  "size": 1,
  "uses_per_turn": 1,
  "dice_condition": {
    "type": "any"
  },
  "effect": {
    "type": "damage",
    "value": "dice_value"
  },
  "tags": ["attack"],
  "upgradable": true
}
```

### 5.3.2 装备尺寸

装备通常占据不同空间：

| 尺寸 | 说明 |
|---|---|
| 小型 | 占 1 格 |
| 中型 | 占 2 格 |
| 大型 | 占 3 格或更多 |

装备容量限制迫使玩家做取舍：

```text
总装备容量 = 角色装备栏容量
当前装备占用 <= 总装备容量
```

例如：

| 装备 | 占用 |
|---|---|
| 短剑 | 1 |
| 盾牌 | 1 |
| 火球 | 2 |
| 大锤 | 2 |

如果角色容量为 4，则可以携带：

```text
短剑 + 盾牌 + 火球 = 1 + 1 + 2 = 4
```

## 5.4 装备条件设计

装备条件是游戏策略的核心。

### 5.4.1 常见条件

| 条件 | 设计效果 |
|---|---|
| 任意骰子 | 稳定，适合基础装备 |
| 偶数骰子 | 需要一定运气，适合中等强度效果 |
| 奇数骰子 | 同上 |
| 最大 $3$ | 消耗低点骰子，避免低点无用 |
| 最小 $4$ | 需要高点数，效果更强 |
| 指定 $6$ | 强效果，但触发不稳定 |
| 总和达到 $N$ | 多骰投入，适合蓄力型装备 |
| 双骰相同 | 高风险高收益 |
| 双骰不同 | 鼓励灵活使用 |
| 递增点数 | 适合连击或技巧流 |

### 5.4.2 条件与效果强度关系

大致可以这样估算：

| 条件难度 | 效果倍率 |
|---|---|
| 任意 | $1.0$ |
| 奇偶 | $1.2$ |
| 小于等于 $3$ | $1.1$ |
| 大于等于 $4$ | $1.2$ |
| 指定点数 | $1.5$ 到 $2.0$ |
| 双骰条件 | $1.8$ 到 $2.5$ |
| 累计大阈值 | $2.0+$ |

## 6.2 回合流程

### 玩家回合

```text
1. 回合开始状态结算
2. 投掷骰子
3. 应用状态影响，如冰冻、燃烧、诅咒
4. 玩家分配骰子
5. 装备效果结算
6. 玩家点击结束回合
```

### 敌人回合

```text
1. 敌人投掷骰子
2. AI 按优先级使用装备
3. 状态效果结算
4. 回合结束
```

---

## 6.3 伤害计算

基础伤害：

$$
Damage = BaseValue
$$

若装备伤害等于骰子点数：

$$
Damage = d
$$

若有倍率：

$$
Damage = d \times M
$$

若目标有护盾：

$$
ActualDamage = \max(0, Damage - Shield)
$$

护盾扣除：

$$
Shield' = \max(0, Shield - Damage)
$$

生命值扣除：

$$
HP' = HP - ActualDamage
$$

---

## 6.4 防御系统

防御可设计为“护盾”，通常只持续当前回合或直到被打破。

```text
使用盾牌装备
    ↓
获得 Block
    ↓
敌人攻击
    ↓
优先扣除 Block
    ↓
剩余伤害扣 HP
```

示例：

玩家有 $5$ 点护盾，敌人造成 $8$ 点伤害：

$$
ActualDamage = 8 - 5 = 3
$$

玩家受到 $3$ 点生命伤害。

# 7. 状态效果系统

状态是丰富构筑的关键。

## 7.1 常见状态

| 状态 | 效果 |
|---|---|
| 毒 | 每回合受到持续伤害 |
| 燃烧 | 某些骰子变为燃烧骰，使用前需支付生命或触发惩罚 |
| 冰冻 | 将最高点数骰子变为 $1$ |
| 麻痹 / 震击 | 锁定一个装备槽，需用骰子解除 |
| 虚弱 | 降低装备效果 |
| 诅咒 | 一定概率导致装备失效 |
| 护盾 | 抵消伤害 |
| 荆棘 | 被攻击时反伤 |
| 沉默 | 禁止某类装备 |
| 中断 | 取消下一个行动 |

## 7.2 状态设计原则

1. **必须视觉明确**
   - 玩家能立刻知道自己被什么状态影响。

2. **要有反制空间**
   - 例如燃烧骰仍然可以用，只是有代价。
   - 冰冻骰仍然变成 $1$，可用于低点数装备。

3. **状态之间应形成流派**
   - 毒伤流：持续伤害。
   - 冰冻流：控制敌人高点骰。
   - 火焰流：高伤害但反噬。
   - 电击流：干扰敌人装备。

# 8. 角色设计

《骰子地下城》的亮点之一是每个角色规则不同。  
这比单纯增加卡牌更有效地增加重玩价值。

## 8.1 角色设计原则

每个角色应有：

1. 独特资源机制
2. 独特装备池
3. 独特成长节奏
4. 独特风险点
5. 独特新手难度

---

## 8.2 角色原型拆解

### 8.2.1 战士型角色

| 项目 | 内容 |
|---|---|
| 定位 | 新手角色 |
| 特点 | 规则直观，攻击、防御清晰 |
| 核心能力 | 可重掷骰子 |
| 强项 | 稳定、容易理解 |
| 弱点 | 后期构筑深度较浅 |

设计用途：

- 教玩家骰子分配。
- 教玩家装备条件。
- 教玩家风险控制。

适合装备：

| 装备类型 | 示例效果 |
|---|---|
| 剑 | 造成骰子点数伤害 |
| 盾 | 获得骰子点数护盾 |
| 大剑 | 需要高点数，造成高伤害 |
| 重掷装备 | 改善骰子质量 |

---

### 8.2.2 盗贼型角色

| 项目 | 内容 |
|---|---|
| 定位 | 灵活变化 |
| 特点 | 可以复制、偷取或使用敌方技能 |
| 核心能力 | 临时获得敌人行动 |
| 强项 | 每场战斗变化大 |
| 弱点 | 稳定性较差 |

设计价值：

- 增加战斗新鲜感。
- 让敌人技能也成为玩家策略的一部分。
- 提高内容复用率。

---

### 8.2.3 机器人型角色

| 项目 | 内容 |
|---|---|
| 定位 | 风险收益 |
| 特点 | 不直接获得骰子，而是通过特殊方式抽取点数 |
| 核心能力 | 类似“二十一点”的超载机制 |
| 强项 | 爆发强 |
| 弱点 | 容易超载或浪费行动 |

设计价值：

- 将骰子随机转化为“抽点数风险”。
- 适合喜欢赌博感的玩家。

---

### 8.2.4 发明家型角色

| 项目 | 内容 |
|---|---|
| 定位 | 装备循环 |
| 特点 | 战斗后装备可能被消耗，转化为临时发明 |
| 核心能力 | 强迫玩家不断更换构筑 |
| 强项 | 每局变化大 |
| 弱点 | 玩家难以长期保留核心装备 |

设计价值：

- 避免玩家固定套路。
- 强化 Roguelike 的适应性。

---

### 8.2.5 女巫型角色

| 项目 | 内容 |
|---|---|
| 定位 | 高策略、高复杂度 |
| 特点 | 将法术放入指定位置，用骰子激活 |
| 核心能力 | 法术书 / 技能栏管理 |
| 强项 | 上限高 |
| 弱点 | 学习成本高 |

设计价值：

- 给高阶玩家深度。
- 通过空间管理和资源规划增加复杂度。

---

### 8.2.6 小丑型角色

| 项目 | 内容 |
|---|---|
| 定位 | 更接近传统卡牌 |
| 特点 | 有抽牌、弃牌、牌组循环机制 |
| 核心能力 | 卡牌组合 |
| 强项 | 连击感强 |
| 弱点 | 初期理解成本较高 |

设计价值：

- 为喜欢传统卡牌构筑的玩家提供角色。
- 丰富游戏整体机制宽度。

# 9. 敌人设计

## 9.1 敌人设计目标

敌人不是单纯数值墙，而是教学和机制展示工具。

每个敌人应承担一个功能：

| 敌人类型 | 教学目的 |
|---|---|
| 普通攻击敌人 | 教基础攻防 |
| 毒敌人 | 教持续伤害 |
| 冰敌人 | 教点数干扰 |
| 火敌人 | 教燃烧代价 |
| 盾敌人 | 教破盾或持续伤害 |
| 高爆发敌人 | 教防御和优先击杀 |
| 治疗敌人 | 教输出节奏 |
| 诅咒敌人 | 教概率风险 |

## 9.3 敌人 AI

敌人 AI 不需要复杂搜索，使用优先级即可。

### AI 优先级示例

```text
1. 如果可以击杀玩家，优先使用伤害装备
2. 如果自身血量低且有治疗，优先治疗
3. 如果玩家没有状态，优先施加状态
4. 使用最高收益伤害技能
5. 使用剩余骰子触发辅助技能
```

### 简单评分模型

每个可行动作计算分数：

$$
Score = DamageValue \times W_d + DefenseValue \times W_b + StatusValue \times W_s + KillBonus
$$

其中：

- $W_d$：伤害权重
- $W_b$：防御权重
- $W_s$：状态权重
- $KillBonus$：击杀奖励分

# 10. 地牢与关卡结构

## 10.1 地牢层级

一局游戏通常由多个楼层组成：

```text
第 1 层：教学层，敌人简单
第 2 层：引入状态机制
第 3 层：开始出现构筑分支
第 4 层：精英敌人和商店压力
第 5 层：高强度组合敌人
第 6 层：Boss
```

## 10.2 地图节点

常见节点：

| 节点 | 功能 |
|---|---|
| 普通战斗 | 主要经验来源 |
| 精英战斗 | 高风险高奖励 |

| 商店 | 购买装备、道具 |
| 铁匠 | 升级装备 |

# 11. 成长系统

## 11.1 经验与升级

战斗后获得经验。升级可获得：

- 最大生命值提升
- 骰子数量增加
- 装备奖励
- 装备栏容量提升
- 特殊技能增强

升级曲线应保持短局节奏。

示例：

| 等级 | 所需经验 | 奖励 |
|---|---:|---|
| 1 | 0 | 初始状态 |
| 2 | 2 | 最大生命 +4 |
| 3 | 5 | 增加 1 个骰子 |
| 4 | 9 | 装备选择 |
| 5 | 14 | 最大生命 +4 |
| 6 | 20 | 最终强化 |

## 11.2 装备奖励

奖励应提供选择，而不是直接给固定物品。

```text
战斗胜利
    ↓
三选一装备
    ↓
玩家选择
    ↓
决定是否装备
```

选择设计：

| 选项 A | 选项 B | 选项 C |
|---|---|---|
| 稳定攻击 | 状态流装备 | 高风险爆发 |

## 11.3 装备升级

升级可以改变：

| 升级方向 | 示例 |
|---|---|
| 降低条件 | 从“偶数”变成“任意” |
| 提高数值 | 伤害从 $d$ 变成 $d+2$ |
| 增加使用次数 | 每回合一次变两次 |
| 降低尺寸 | 占用从 2 变 1 |
| 增加附加效果 | 攻击同时施加状态 |
| 降低阈值 | 累计 $12$ 点变成 $10$ 点 |

# 18. 平衡设计

## 18.1 骰子期望值

普通六面骰期望值：

$$
E(d) = \frac{1+2+3+4+5+6}{6} = 3.5
$$

如果玩家每回合有 $n$ 个骰子，则总点数期望为：

$$
E(total) = 3.5n
$$

例如 3 个骰子：

$$
E(total) = 10.5
$$

这可以作为装备平均输出设计的基础。

---

## 18.2 装备价值估算

假设装备每回合一次，任意骰子造成点数伤害，则平均伤害：

$$
AvgDamage = 3.5
$$

如果装备要求偶数，则触发概率为：

$$
P(even) = \frac{3}{6} = 0.5
$$

但玩家通常有多个骰子，实际可触发概率更高。

如果有 $n$ 个骰子，至少一个偶数的概率是：

$$
P = 1 - \left(\frac{1}{2}\right)^n
$$

例如 3 个骰子：

$$
P = 1 - \frac{1}{8} = 0.875
$$

所以“偶数条件”在多骰情况下并没有看起来那么难，需要谨慎提高数值。

---

## 18.3 敌人强度估算

敌人每回合威胁可由以下组成：

$$
Threat = Damage + StatusValue + DefenseValue
$$

其中状态价值可以换算为伤害等价。

例如：

| 状态 | 可估价值 |
|---|---:|
| 1 毒 | 约等于 1-2 点未来伤害 |
| 1 冰冻 | 约等于降低玩家 2-4 点资源价值 |
| 1 燃烧 | 约等于 1-2 点压力 |
| 1 震击 | 约等于限制一次行动 |

---

## 18.4 战斗长度

普通战斗建议控制在：

$$
3 \leq Turns \leq 6
$$

Boss 战建议：

$$
6 \leq Turns \leq 10
$$

如果战斗太短，构筑感无法体现。  
如果战斗太长，轻量体验会变拖沓。

# 22. 伪代码参考

## 22.1 回合开始

```pseudo
function startTurn(unit):
    applyStartTurnStatus(unit)

    dice = rollDice(unit.diceCount)

    dice = applyDiceModifiers(unit, dice)

    unit.availableDice = dice

    resetEquipmentUses(unit)
```

## 22.2 使用装备

```pseudo
function useEquipment(unit, equipment, dice):
    if equipment.usedThisTurn >= equipment.usesPerTurn:
        return false

    if not checkDiceCondition(equipment.condition, dice):
        return false

    consumeDice(unit, dice)

    resolveEffect(unit, equipment.effect, dice)

    equipment.usedThisTurn += 1

    return true
```

## 22.3 条件判断

```pseudo
function checkDiceCondition(condition, dice):
    value = dice.value

    switch condition.type:
        case "any":
            return true

        case "even":
            return value % 2 == 0

        case "odd":
            return value % 2 == 1

        case "max":
            return value <= condition.value

        case "min":
            return value >= condition.value

        case "exact":
            return value == condition.value

        case "sum":
            return sum(dice.values) >= condition.value

        case "pair":
            return dice.length == 2 and dice[0].value == dice[1].value
```

## 22.4 效果结算

```pseudo
function resolveEffect(user, effect, dice):
    target = getTarget(effect.target)

    if effect.type == "damage":
        value = calculateValue(effect.value, dice)
        dealDamage(target, value)

    if effect.type == "shield":
        value = calculateValue(effect.value, dice)
        user.shield += value

    if effect.type == "poison":
        target.status.poison += effect.value

    if effect.type == "heal":
        user.hp = min(user.maxHp, user.hp + effect.value)
```

# 24. 开发风险

## 24.1 随机性过强

问题：

- 玩家觉得输是因为骰子差。
- 决策感下降。

解决：

- 增加重掷、变点、复制、保留骰子的装备。
- 提供低点数也有用的装备。
- 避免大量“必须指定 $6$”的装备。

## 24.2 装备池污染

问题：

- 装备太多导致玩家很难拿到核心构筑。
- 随机奖励体验变差。

解决：

- 按角色限制装备池。
- 按标签生成奖励。
- 提供重选奖励。
- 商店提供定向补强。

## 24.3 角色复杂度失控

问题：

- 每个角色机制都不同，开发和测试成本大。

解决：

- 先做基础角色。
- 其他角色复用核心系统，只改资源规则。
- 每个角色只引入一个核心变化。

## 24.4 平衡成本高

问题：

- 装备、敌人、角色、章节组合爆炸。

解决：

- 所有装备数据表化。
- 自动模拟战斗。
- 记录玩家胜率。
- 按章节分装备池。
- 先平衡普通难度，再做挑战难度。

# 25. 总结

《骰子地下城》的核心成功点不是“骰子随机”，而是：

1. **把骰子变成了可分配资源。**
2. **把装备变成了固定手牌。**
3. **用角色机制替代复杂卡池。**
4. **用短局 Roguelike 保持重玩性。**
5. **让每个点数都有潜在价值。**

如果你要开发参考版，建议优先实现：

```text

装备条件
装备效果
回合制战斗
状态效果
战斗奖励
地图节点
角色成长
```
