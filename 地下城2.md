
# 22. 装备设计扩展

装备是本类游戏最核心的内容资源之一。  
它既决定了玩家每回合的行动方式，也决定了构筑方向、角色差异和战斗节奏。

---

## 22.1 装备设计目标

优秀的装备应满足以下至少一项：

1. 提供稳定输出
2. 提供资源转化
3. 利用特定点数骰子
4. 提供状态流派入口
5. 提供爆发机会
6. 提供防御或回复
7. 改变骰子本身
8. 与其他装备形成组合

---

## 22.2 装备分类

### 22.2.1 攻击类装备

攻击类装备负责造成直接伤害。

| 装备类型 | 示例效果 | 设计定位 |
|---|---|---|
| 基础攻击 | 造成骰子点数伤害 | 稳定输出 |
| 固定伤害 | 偶数骰，造成 6 点伤害 | 中等稳定 |
| 高点爆发 | 最小 5，造成 10 点伤害 | 奖励高点数 |
| 多段攻击 | 造成 2 次骰子点数伤害 | 适合增益配合 |
| 穿透攻击 | 无视护盾造成伤害 | 针对防御敌人 |
| 随机攻击 | 造成 1-12 点伤害 | 风险收益 |

### 22.2.2 防御类装备

防御类装备用于降低伤害、延长战斗时间。

| 装备类型 | 示例效果 | 设计定位 |
|---|---|---|
| 护盾 | 获得骰子点数护盾 | 基础防御 |
| 固定护盾 | 最大 3，获得 6 护盾 | 利用低点数 |
| 反伤 | 获得护盾并反弹伤害 | 防守反击 |
| 闪避 | 下次攻击伤害减半 | 高价值防御 |
| 净化 | 移除 1 个负面状态 | 状态反制 |
| 回复 | 恢复生命 | 稀缺生存资源 |

### 22.2.3 骰子操控类装备

骰子操控类装备用于降低随机性。

| 装备类型 | 示例效果 | 设计价值 |
|---|---|---|
| 重掷 | 重掷一个骰子 | 提高稳定性 |
| 加点 | 一个骰子 $+1$ | 达成阈值 |
| 减点 | 一个骰子 $-1$ | 达成低点条件 |
| 翻面 | 将骰子变为 $7-d$ | 点数转换 |
| 复制 | 复制一个骰子 | 爆发或组合 |
| 拆分 | 将一个骰子拆成两个骰子 | 增加行动次数 |
| 合并 | 合并两个骰子 | 获取高点数 |

### 22.2.4 状态类装备

状态类装备决定战斗流派。

| 状态 | 装备设计方向 |
|---|---|
| 毒 | 叠加持续伤害 |
| 冰冻 | 干扰敌方骰子点数 |
| 燃烧 | 提高伤害但附带风险 |
| 震击 | 锁定敌人装备 |
| 虚弱 | 降低敌人输出 |
| 诅咒 | 让敌人行动有概率失败 |
| 易伤 | 增加后续伤害 |
| 流血 | 攻击时追加伤害 |

---

## 22.3 装备强度估算

假设一个装备每回合可以使用一次，且接受任意骰子。

如果效果是造成骰子点数伤害，则平均伤害为：

$$

AvgDamage = 3.5

$$

如果装备造成骰子点数 $+2$ 的伤害，则平均伤害为：

$$

AvgDamage = 3.5 + 2 = 5.5

$$

如果装备要求偶数，单骰触发概率为：

$$

P(even)=0.5

$$

但如果玩家有 3 个骰子，至少出现一个偶数的概率为：

$$

P = 1 - \left(\frac{1}{2}\right)^3 = 0.875

$$

因此，奇偶限制不应被过度高估。

# 23. 状态效果详细设计

状态效果应作为构筑深度的重要来源。  
每种状态都应有清晰的数值模型、触发时机和反制方式。

---

## 23.1 状态通用字段

```json
{
  "status_id": "poison",
  "name": "毒",
  "stack": 3,
  "duration": -1,
  "trigger_timing": "turn_start",
  "is_positive": false,
  "can_stack": true,
  "can_cleanse": true
}
```

| 字段 | 说明 |
|---|---|
| status_id | 状态唯一 ID |
| name | 状态名称 |
| stack | 层数 |
| duration | 持续回合，-1 表示特殊规则 |
| trigger_timing | 触发时机 |
| is_positive | 是否为正面状态 |
| can_stack | 是否可叠加 |
| can_cleanse | 是否可被清除 |

---

## 23.2 毒

### 效果

每回合开始时，受到等同于毒层数的伤害，然后毒层数减少 1。

例如当前有 4 层毒：

```text
回合开始受到 4 点伤害
毒层数变为 3
```

### 公式

第 $t$ 回合毒伤为：

$$

Damage_t = PoisonStack_t

$$

若每回合减少 1 层，则总伤害为：

$$

TotalDamage = \frac{n(n+1)}{2}

$$

例如 4 层毒的总伤害为：

$$

TotalDamage = \frac{4 \times 5}{2} = 10

$$

### 设计定位

- 适合持久战
- 对高血量敌人有效
- 前期收益较慢，后期收益较高

---

## 23.3 冰冻

### 效果

使目标下回合最高点数骰子变为 1。

例如敌人投出：

```text
6, 4, 2
```

受到 1 层冰冻后变为：

```text
1, 4, 2
```

### 多层冰冻

若有多层冰冻，则依次冻结最高点数骰子。

```text
原始骰子：6, 5, 3
冰冻 2 层后：1, 1, 3
```

### 设计定位

- 削弱高点数装备
- 对依赖高点敌人很强
- 对低点装备流派敌人较弱

---

## 23.4 燃烧

### 效果

随机将若干骰子变为燃烧骰。  
燃烧骰仍可使用，但使用时受到 2 点伤害，或必须先支付代价。

### 设计价值

燃烧不是禁止玩家行动，而是制造选择：

```text
是否为了打出关键装备而承受伤害？
```

### 设计定位

- 制造风险收益
- 适合火焰流敌人
- 不应让玩家完全无法行动

---

## 23.5 震击

### 效果

锁定目标一个装备槽。  
被震击的装备不能直接使用，必须先投入一个骰子解除。

### 设计定位

- 限制玩家行动顺序
- 增加骰子消耗
- 对关键装备有明显压制力

### 注意事项

震击过多会导致玩家体验非常差。  
建议限制：

```text
普通敌人每回合最多施加 1-2 层震击
Boss 最多施加 2-3 层震击
```

---

## 23.6 诅咒

### 效果

玩家使用装备时，有一定概率使该装备失效。

例如：

$$

P(fail)=50\%

$$

### 设计风险

诅咒属于高挫败状态。  
如果玩家关键回合失败，可能觉得不公平。

### 解决方案

1. 显示明确概率
2. 允许净化
3. 限制持续时间
4. 避免连续多次失败
5. 可加入伪随机保护

# 24. 事件系统设计

事件系统用于丰富地牢探索，让玩家在战斗之外做选择。

---

## 24.1 事件设计目标

事件应提供：

1. 风险与收益
2. 短期与长期选择
3. 构筑方向调整
4. 资源交换
5. 世界观表达

---

## 24.2 事件类型

| 类型 | 示例 |
|---|---|
| 资源交换 | 失去生命，获得金币 |
| 装备变化 | 升级一件随机装备 |
| 构筑选择 | 获得毒系装备或火系装备 |
| 风险战斗 | 立即挑战精英，获得稀有奖励 |
| 负面事件 | 获得诅咒，但获得强力装备 |
| 商人事件 | 低价购买特殊装备 |
| 治疗事件 | 回复生命或提高最大生命 |

1. 购买装备

3. 回复生命
4. 升级装备

|---|---:|
| 普通装备 | 5-8 金币 |

```text

第二次刷新：4 金币

```

价格递增可以防止玩家无限寻找核心装备。

# 26. Boss 设计

Boss 是对玩家当前构筑的最终检验。

---

## 26.1 Boss 设计原则

Boss 应该：

1. 有独特机制
2. 不只是高血量敌人
3. 考验玩家构筑完整性
4. 给玩家足够反应空间
5. 避免不可反制的秒杀
6. 战斗时间控制在 6-10 回合

---

## 26.2 Boss 类型

| 类型 | 特点 |
|---|---|
| 高爆发 Boss | 检验玩家防御能力 |
| 状态 Boss | 检验净化和节奏 |
| 召唤 Boss | 检验群体处理能力 |
| 护盾 Boss | 检验持续输出或穿透 |
| 骰子干扰 Boss | 检验骰子操控能力 |
| 多阶段 Boss | 检验长期资源管理 |

---

## 26.3 Boss 样例：钟表巨像

### 设计定位

- 骰子点数干扰型 Boss
- 主题是时间、倒计时、节奏控制

### 数据示例

```json
{
  "id": "boss_clockwork_colossus",
  "name": "钟表巨像",
  "hp": 72,
  "dice_count": 3,
  "equipment": [
    "gear_punch",
    "time_lock",
    "reverse_dice",
    "countdown_beam"
  ],
  "tags": ["boss", "dice_control"]
}
```

### 技能设计

| 技能 | 条件 | 效果 |
|---|---|---|
| 齿轮重击 | 任意 | 造成骰子点数 +2 伤害 |
| 时间锁 | 偶数 | 震击 1 个装备 |
| 逆转指针 | 奇数 | 将玩家一个骰子翻面 |
| 倒计时光束 | 累计 12 | 造成 16 点伤害 |

### 设计意图

- 玩家要判断何时防御
- 玩家要保留净化或解震击能力
- Boss 的累计技能制造回合压力

# 27. 地图生成设计

地图结构决定玩家的路线选择。

---

## 27.1 地图生成目标

地图应提供：

1. 可读性
2. 路线分支
3. 风险收益选择
4. 不可完全预测的随机性
5. 足够但不过量的节点

---

## 27.2 地图节点权重

不同楼层可使用不同节点权重。

| 节点 | 第 1 层 | 第 2-3 层 | 第 4-5 层 |
|---|---:|---:|---:|
| 普通战斗 | 70% | 50% | 40% |
| 精英战斗 | 0% | 10% | 20% |
| 宝箱 | 10% | 10% | 10% |
| 商店 | 5% | 10% | 10% |
| 治疗 | 10% | 10% | 10% |
| 事件 | 5% | 10% | 10% |

---

## 27.3 地图生成伪代码

```pseudo
function generateFloor(floorIndex):
    nodes = []

    nodeCount = getNodeCountByFloor(floorIndex)

    for i in range(nodeCount):
        nodeType = weightedRandom(getNodeWeights(floorIndex))
        nodes.append(createNode(nodeType))

    ensureAtLeastOneBattle(nodes)

    if floorIndex > 1:
        ensureOptionalShopOrHeal(nodes)

    return connectNodes(nodes)
```

---

## 27.4 路线连接原则

1. 每个节点至少有一个后续节点。
2. 不要出现明显唯一最优路线。
3. 精英战斗后应有较高概率接奖励节点。
4. 治疗节点不应过密。
5. 商店前应保证玩家有机会获得金币。

# 28. 解锁与长期进度

虽然游戏以单局 Roguelike 为核心，但长期目标能显著提升留存。

---

## 28.1 可解锁内容

| 内容 | 解锁方式 |
|---|---|
| 新角色 | 击败指定 Boss |
| 新装备 | 完成角色章节 |
| 新章节 | 使用角色通关 |
| 新难度 | 多角色通关 |
| 外观 | 达成成就 |
| 挑战模式 | 完成基础内容 |
| 图鉴 | 遇到或击败敌人 |

---

## 28.2 解锁设计原则

1. 不应让初始内容太少。
2. 新角色应尽早解锁。
3. 强力装备不应完全依赖长线解锁。
4. 解锁内容应增加玩法宽度，而不是单纯数值增强。
5. 失败也应有轻微进度感。

---

## 28.3 成就设计样例

| 成就 | 条件 |
|---|---|
| 完美防御 | 单场战斗不受到生命伤害 |
| 点数大师 | 一回合使用 6 个骰子 |
| 毒性扩散 | 对敌人叠加 10 层毒 |
| 极限翻盘 | 生命低于 3 时击败 Boss |
| 节俭冒险者 | 不进入商店完成一局 |
| 高速通关 | 20 分钟内通关 |

# 29. 新手引导设计

轻量游戏也需要引导，尤其是骰子条件和装备结算。

---

## 29.1 教学目标

新手前 5 分钟应学会：

1. 投掷骰子
2. 拖拽骰子
3. 使用装备
4. 理解敌人攻击
5. 结束回合
6. 获得奖励
7. 替换装备

---

## 29.2 教学流程

```text
第一场战斗：
只给玩家一个攻击装备
教学：拖骰子造成伤害

第二场战斗：
加入盾牌
教学：防御与敌人回合

第三场战斗：
加入条件装备
教学：偶数、最大值、指定点数

第一次奖励：
教学：选择装备

第一次升级：
教学：角色成长
```

---

## 29.3 引导原则

1. 不要一次弹出大量文字。
2. 用高亮提示交互区域。
3. 允许玩家跳过。
4. 失败后重复关键提示。
5. 用实际操作代替说明书。

# 30. 数据驱动架构建议

为了便于平衡和扩展，建议尽量数据驱动。

---

## 30.1 推荐模块划分

```text
Core
 ├── DiceSystem
 ├── CombatSystem
 ├── EquipmentSystem
 ├── StatusSystem
 ├── EnemyAI
 ├── RewardSystem
 ├── MapSystem
 ├── ShopSystem
 └── SaveSystem
```

---

## 30.2 装备效果配置化

推荐将效果拆为条件和动作。

```json
{
  "id": "fire_blade",
  "name": "火刃",
  "condition": {
    "type": "even"
  },
  "effects": [
    {
      "type": "damage",
      "target": "enemy",
      "value": 6
    },
    {
      "type": "apply_status",
      "target": "enemy",
      "status": "burn",
      "value": 1
    }
  ]
}
```

这种结构允许一个装备拥有多个效果。

---

## 30.3 效果执行器

```pseudo
function executeEffects(user, target, effects, dice):
    for effect in effects:
        switch effect.type:
            case "damage":
                dealDamage(target, evaluate(effect.value, dice))

            case "shield":
                addShield(user, evaluate(effect.value, dice))

            case "apply_status":
                addStatus(target, effect.status, effect.value)

            case "modify_dice":
                modifyDice(user, effect.modifier)
```

---

## 30.4 表达式系统

为提升配置灵活性，可以支持表达式：

```json
{
  "type": "damage",
  "value": "dice * 2 + 1"
}
```

解析后：

$$

Damage = d \times 2 + 1

$$

注意表达式系统要限制能力，避免执行任意代码。

# 32. 平衡测试与自动模拟

---

## 32.1 为什么需要自动模拟

随着装备和敌人变多，人工测试无法覆盖全部组合。

需要模拟：

- 不同角色胜率
- 不同装备强度
- 不同敌人威胁
- 不同楼层死亡率
- 不同路线收益

---

## 32.2 模拟指标

| 指标 | 说明 |
|---|---|
| Win Rate | 通关率 |
| Avg Turn Count | 平均战斗回合数 |
| Avg Damage Taken | 平均受伤 |
| Equipment Pick Rate | 装备选择率 |
| Equipment Win Rate | 拿到装备后的胜率 |
| Floor Death Rate | 各层死亡率 |
| Boss Kill Rate | Boss 击败率 |

---

## 32.3 简单模拟 AI

```pseudo
function playerSimulationTurn(state):
    if canKillEnemy(state):
        useBestDamageActions(state)
    else:
        if incomingDamageHigh(state):
            useDefenseActions(state)

        useBestExpectedDamageActions(state)

        useRemainingUtilityActions(state)
```

---

## 32.4 平衡目标参考

| 指标 | 目标值 |
|---|---:|
| 新手角色普通难度通关率 | 40%-60% |
| 高阶角色普通难度通关率 | 30%-50% |
| 普通战斗平均回合 | 3-6 |
| Boss 战平均回合 | 6-10 |
| 第 1 层死亡率 | 低于 5% |
| 最后一层死亡率 | 20%-35% |

2. 骰子池共享
3. 敌人可以攻击不同目标

| 阵位 | 特点 |
|---|---|

| 中排 | 平衡位置 |

# 43. 结语扩展

这类“骰子 + 装备 + Roguelike”的轻量策略游戏，最重要的是让玩家持续感受到：

```text
我虽然掷出了随机结果，
但我仍然可以通过选择改变结果。
```

因此，设计重点不是完全消除随机，而是让随机成为策略的一部分。

一个好的回合应该让玩家产生类似这样的思考：

```text
我现在有 6、3、1。
6 可以打爆发，也可以拆成两个骰子。
3 可以施毒，也可以加点变成 4。
1 看起来很差，但刚好可以触发低点装备。
如果我这回合防御，下回合毒就能杀死敌人。
```

当每个点数都有价值，每个装备都有适用场景，每次路线选择都有代价时，游戏就会形成足够的重玩深度。

最终设计目标可以总结为：

1. 骰子提供变化。
2. 装备提供选择。
3. 状态提供流派。
4. 地图提供取舍。
5. 角色提供重玩。
6. Boss 提供检验。
7. 解锁提供长期目标。

```text
随机不是敌人。
无选择的随机才是敌人。
```
